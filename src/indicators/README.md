# 지표 산출 파이프라인 (src/indicators/) — 4개 파일이 하는 일

## 큰 그림 먼저

이 4개 파일은 두 가지 목표를 위해 순서대로 실행된다.

> **① 청년 1인가구 실거래가 데이터에 외부 공식 통계(한국부동산원 전월세전환율)를 붙여서 행정동별 표면 주거비를 산출하고, ② 서울시 상권분석서비스 데이터로 행정동별 생활소비부담지수를 산출한 뒤, 이 둘을 하나로 결합하는 것**

표면 주거비 산식은 처음부터 정해져 있었다.

```
표면 주거비 = 실제 월세 + (실제 보증금 × 전월세전환율 ÷ 12)
```

문제는 `전월세전환율`이 우리 데이터 어디에도 없는 **외부 지표**라는 점이었다. 이걸 구해서 붙이는 과정이 ①②③, 그리고 생활소비부담지수를 산출해 결합하는 과정이 ④다.

```
① load_conversion_rate.py       (단독다가구·연립다세대 전환율 정리)
        ↓
② load_officetel_rate.py        (오피스텔 전환율 정리, 별도 파일인 이유는 구조가 다르기 때문)
        ↓
③ calc_surface_housing_cost.py  (실거래가 데이터에 전환율을 붙여 표면 주거비 계산 + 행정동별 집계)
        ↓
④ calc_consumption_burden_index.py  (생활소비부담지수 산출 + 표면 주거비와 결합)
```

---

## ① load_conversion_rate.py — 단독다가구·연립다세대 전환율 정리

**왜 필요했나**: 표면 주거비 산식에 넣을 전월세전환율을 어디서 구할지부터 막혔다. 처음엔 "구별로 다르게 할지, 서울 전체 단일값으로 할지"를 고민했는데, 알아본 결과 이 결정 자체가 무의미했다 — 서울시가 발표하던 자치구별·주택유형별 통계(서울 열린데이터광장)는 **2022년에 서비스가 종료**됐고, 한국부동산원의 예전 권역별 통계도 **2018년 1분기로 수록이 끝나** 있었다. 즉 "얼마나 세밀하게 쓸지"보다 먼저 "지금도 갱신되는 통계가 있는지"부터 찾아야 했다.

**최종적으로 찾은 데이터**: 한국부동산원 R-ONE의 "지역별 전월세 전환율" — 단독주택/연립다세대/아파트/종합주택 4종류로 나뉘어 있고, **서울시 5개 권역(도심권/동남권/동북권/서남권/서북권) 단위로 월별 최신 데이터**까지 살아있었다. 자치구 단위는 아니지만, 서울 전체 단일값보다는 훨씬 정밀하다.

**문제였던 점**: 다운받은 표에 "강북지역"/"강남지역"이라는 항목이 섞여 있었는데, 이건 실제 권역이 아니라 **다른 권역들을 합친 요약값**이었다(강북지역 = 도심권+동북권+서북권, 강남지역 = 동남권+서남권). 그대로 뒀으면 나중에 집계할 때 같은 자치구 데이터가 이중으로 잡힐 뻔했다.

**하는 일**: 단독주택/연립다세대 CSV 두 개에서 강북·강남 요약행을 빼고 실제 5개 권역만 남긴 뒤, 가로형(월별 컬럼)을 세로형(연월 하나의 컬럼)으로 바꾼다. 그리고 "자치구 → 권역" 매핑표도 별도로 만들어둔다 — 우리 실거래가 데이터는 자치구 단위인데 전환율은 권역 단위라서, 나중에 이 둘을 이어줄 다리가 필요하기 때문이다.

**입력 → 출력**: `지역별_전월세_전환율_단독주택.csv`, `지역별_전월세_전환율_연립_다세대.csv` → `전월세전환율_권역별.csv`, `자치구_권역_매핑.csv`

## ② load_officetel_rate.py — 오피스텔 전환율 정리 (별도 파일인 이유)

**왜 따로 만들었나**: 오피스텔은 애초에 다른 통계(오피스텔 가격동향조사)에서 나온다. 처음엔 "오피스텔은 시군구 선택이 안 되는 조사라 서울 전체 단일값만 쓸 수 있겠다"고 판단했는데, 실제로 받아보니 예상보다 훨씬 정밀했다 — **권역별로도 나뉘어 있고, 거기에 더해 면적구간(40㎡이하 / 40㎡초과~60㎡이하)까지 세분화**되어 있었다. 구조 자체가 단독주택·연립다세대 표와 다르기 때문에(권역만 있는 게 아니라 면적구간이라는 축이 하나 더 있음), 같은 로더로 처리할 수 없어서 별도 파일로 분리했다.

**하는 일**: 오피스텔 CSV에서 서울 5개 권역 × 2개 면적구간(전체 60㎡ 이하 데이터만 다루므로 이 두 구간이면 충분) 조합만 남기고, 마찬가지로 세로형으로 변환한다.

**입력 → 출력**: `오피스텔_전월세전환율.csv` → `전월세전환율_오피스텔_권역별.csv`

## ③ calc_surface_housing_cost.py — 실제 계산 + 행정동별 집계

**왜 ①②와 분리했나**: ①②는 "외부 통계 정리"고, ③은 "우리 데이터에 그걸 적용해서 계산"하는 완전히 다른 작업이다. 원본 CSV 파싱 로직과 계산 로직을 분리해두면, 나중에 산식이 바뀌거나 필터링 기준이 바뀔 때 ③만 다시 돌리면 되고, 전환율 원본을 매번 다시 파싱할 필요가 없다.

**하는 일**:
1. 청년 1인가구 실거래가 데이터의 `시군구명`(자치구)을 매핑표로 **권역**으로 변환
2. 단독다가구·연립다세대는 (주택유형, 권역, 연월) 기준으로, 오피스텔은 (권역, 면적구간, 연월) 기준으로 전환율을 붙인다
3. 전환율 데이터가 없는 연월(수집 기간 밖)은 가장 가까운 달의 값으로 대체하고 `전환율_범위외_보정` 컬럼에 표시
4. 거래 1건 단위로 표면 주거비를 계산한 뒤(집계값끼리 계산하면 안 된다는 원칙은 3주차부터 계속 지켜온 것과 동일), 행정동 × 주택유형별로 **중앙값**을 집계

**QA 과정에서 발견하고 고친 문제 1 (필터링 기준 문제)**: 계산 결과의 최솟값이 비정상적으로 낮게 나와서(1.475만원) 확인해보니, `filter_youth_household.py`에 월세=0인 행이나 "월세 1만원 + 보증금 100만원"처럼 **자릿수가 빠진 것으로 보이는 입력 오류 행**이 걸러지지 않고 남아있었다. 월세 상한을 임의의 숫자 대신 분포의 상위 1%로 객관적으로 정했던 것과 같은 논리를, 하한에도 대칭으로 적용해서 해결했다 (하위 1% = 10만원을 컷오프로 사용).

**QA 과정에서 발견하고 고친 문제 2 (행정동명 캐시 오염)**: `표면주거비_행정동별.csv`의 `행정동명_최종` 값 일부가 "서울특별시 강남구 개포2동"처럼 시/구가 붙은 채로 섞여 있었다. 표면 주거비 계산 단계가 아니라 그 이전 법정동→행정동 매칭 단계(`resolve_precise_jibun.py`)의 캐시 오염 문제였다. 자세한 원인과 전체 해결 과정은 아래 "디버깅 기록" 섹션 참고.

**입력 → 출력**: `전월세_실거래가_청년1인가구.csv` + `전월세전환율_권역별.csv` + `전월세전환율_오피스텔_권역별.csv` + `자치구_권역_매핑.csv` → `표면주거비_거래단위.csv`(QA용), `표면주거비_행정동별.csv`(최종 산출물)

**결과**: 577,761 / 577,761행(100%) 계산 성공.

## ④ calc_consumption_burden_index.py — 생활소비부담지수 산출 + 결합

**왜 필요했나**: 표면 주거비만으로는 "그 동네에 살 때 매달 나가는 밥값·생필품비" 같은 생활물가 체감도를 반영하지 못한다. 청년이 특정 행정동에 살 때 숨만 쉬어도 나가는 외식비·마트 비용 등 "동네 장바구니 물가"를 정량화한 보조 지표가 필요했다.

**산식 원칙**: 팀원이 작성한 최초 기획안의 산식(모든 업종을 섞어서 평균)에 오류가 있었다 — 업종을 안 나누고 평균을 내면 거래건수가 많은 업종(편의점 등)이 결과를 지배해버린다. 그래서 원 기획서의 진짜 산식대로 **업종별로 먼저 z-score를 계산한 뒤, 행정동별로 그 z-score들을 평균**내는 순서로 수정했다.

**하는 일**:
1. 서울시 상권분석서비스 추정매출 데이터(2023~2026년 4개 파일, 분기별)를 통합. 파일마다 수집 기간이 롤링되어(예: "2026년" 파일에 2025년 분기까지 포함) 겹치는 경우가 있어 (분기, 행정동, 업종) 기준으로 중복 제거
2. 청년 생활밀접 업종 10개(한식·중식·일식·제과·분식·치킨·패스트푸드·커피-음료·슈퍼마켓·편의점)만 필터링
3. 건당 결제금액(매출금액÷매출건수) 산출 후, 매출건수 하위 5%(표본이 너무 적어 단가가 왜곡된 행)를 분위수 기준으로 제거
4. 업종별 z-score 계산 → 행정동별 평균 → 생활소비부담지수 확정
5. 표면 주거비(`행정동명_최종`)와 생활소비부담지수(`행정동_코드_명`)를 행정동명 기준으로 결합

**QA 과정에서 발견하고 고친 문제**: 결합 시 매칭 실패가 반복적으로 발생했다. 원인은 행정동명 표기 불일치였고, 이 문제를 추적·해결한 전체 과정은 아래 "디버깅 기록" 섹션에 시간 순서대로 정리했다.

**입력 → 출력**: `서울시_상권분석서비스_추정매출-행정동_*.csv`(4개) + `표면주거비_행정동별.csv` + `행정동_공식명_목록.csv` → `생활소비부담지수_행정동별.csv`, `표면주거비_생활소비부담지수_결합.csv`(최종 결합본)

**결과**: 1,144개 조합 중 1,132개(98.9%) 매칭 성공.

## 디버깅 기록 — 행정동명 표기 불일치 버그 (시간 순서)

### 매칭 실패란 무엇이고 왜 생기는가 (쉬운 설명)

표면 주거비(청년 1인가구 실거래가 기반)와 생활소비부담지수(서울시 상권분석서비스 기반)는 **완전히 다른 두 기관의 데이터**다. 하나로 합치려면 "행정동 이름"이 글자 하나까지 정확히 똑같아야 컴퓨터가 같은 동으로 인식한다. 문제는 같은 동을 부르는 이름이 기관마다, 심지어 API마다 조금씩 다르다는 것이다.

- 어떤 곳은 "가양1동"이라고 쓰고, 어떤 곳은 "가양**제**1동"이라고 쓴다 (둘 다 진짜 존재하는 표기 방식)
- 어떤 곳은 "면목3.8동"이라고 쓰고, 어떤 곳은 "면목제3.8동"이라고 쓴다
- 심지어 파일 하나가 인코딩 손상으로 "금호2.3가동"의 마침표(.)가 물음표(?)로 깨져 저장되기도 한다

사람 눈에는 "그게 그거지"라고 넘어갈 수 있지만, 컴퓨터는 문자 하나만 달라도 완전히 다른 값으로 취급해서 "매칭 실패"로 처리한다. 아래 기록은 이 "같은 동인데 다르게 적힌" 케이스들을 하나의 기준(국토지리정보원 KIKmix 공식 표기)으로 통일해나간 과정이다.

### 1. 발단 — 표면주거비 결과에 이상한 값 발견
`표면주거비_행정동별.csv`의 544행부터 `행정동명_최종` 값이 "중화제2동"이 아니라 "서울특별시 중랑구 중화제2동"처럼 시/도·구가 붙어 나오는 것을 발견. 원인은 `kakao_cache.json`이 preprocessing 단계 여러 곳에서 이어서 갱신되는 구조인데, **API 응답 검증 로직을 추가하기 전의 예전 값이 캐시에 남아있었던 것**. `normalize_cache.py`로 캐시를 일괄 정리하고, `resolve_precise_jibun.py`의 최종 저장 직전에 정규화 안전장치(시/구 접두어 제거)를 추가해 해결.

### 2. 생활소비부담지수 작업 시작 — 산식 오류 발견
기획안(PDF)을 검토하다가, Step2의 산식이 원래 기획서("업종별 z-score 평균")와 다르게 **업종을 안 나누고 다 섞어서 평균**을 내는 방식으로 되어 있는 것을 발견. "업종별 z-score → 평균" 순서로 수정. 업종 목록도 원안 6개(한식/중식/일식/제과/커피음료/슈퍼마켓/편의점)에서 분식·치킨·패스트푸드 3개를 추가해 10개로 확정.

### 3. 첫 결합 시도 — 매칭 실패 371건
`표면주거비_행정동별.csv`(1,470행)와 생활소비부담지수를 `행정동명_최종` 기준으로 합쳤더니 371건이 매칭 실패. 실패 목록이 전부 "가양제1동", "구로제1동"처럼 "제"가 붙은 행정동이었음.

### 4. 원인 조사 — "제" 표기 불일치 발견 (169쌍)
`표면주거비_거래단위.csv` 안에서 "제"가 없는 동(262개)과 있는 동(180개)을 비교해보니, **같은 동인데 두 형태로 쪼개져 존재하는 경우가 169쌍**이나 발견됨. 원인: 법정동→행정동 매칭 시 텍스트 매칭 경로(KIKmix 원본, "제" 있음)와 API 재조회 경로(카카오/juso, "제" 생략)가 서로 다른 표기를 반환하고 있었음.

### 5. 1차 수정 — 정규화 함수 추가
`dong_matcher.py`가 KIKmix 코드표에서 "공식 행정동명 목록"을 뽑아 별도 파일(`행정동_공식명_목록.csv`)로 저장하고, `resolve_precise_jibun.py`의 최종 출력 단계에서 이 목록과 대조해 "제" 유무를 자동으로 되돌리는 `normalize_dong_name()` 함수를 추가.

### 6. (곁가지) 경로 설정 버그 재발
편집 과정에서 실수로 여러 스크립트(`housing_preprocessor.py`, `resolve_ambiguous.py`, `juso_resolver.py`)의 `PROJECT_ROOT` 경로 깊이가 `parent.parent`(2단계)로 되돌아가 있는 것을 발견 (`src/preprocessing/` 폴더로 옮긴 뒤로는 `parent.parent.parent`가 맞음). 전체 파일 grep으로 일괄 점검 후 수정.

### 7. (곁가지) housing_preprocessor.py의 오작동
"오피스텔"로 시작하는 모든 파일을 원본 실거래가 파일로 착각해서, 오늘 새로 만든 `오피스텔_전월세전환율.csv`까지 잘못 삼켜버리는 버그 발견. 파일명에 "실거래가"라는 단어가 포함되어야 한다는 조건을 추가로 걸어 해결.

### 8. 재검증 — 여전히 432개 (목표 426개 초과)
1차 수정을 반영해 다시 돌렸더니 정규화 후 고유 행정동이 595개→432개로 줄었지만, 목표(서울 행정동 수 426개)보다 여전히 많았음. 원인 진단: `dong_matcher.py`가 "공식 목록"을 만들 때 **분동된 법정동마다 대표값 하나만 골라놓은 `lookup` 결과**를 썼는데, 대표로 안 뽑힌 진짜 동(가양2동, 가양3동 등)이 통째로 목록에서 빠져 있었음. `code_df`(코드표 원본) 전체의 읍면동명을 쓰도록 수정.

### 9. 재검증 — 427개 (거의 근접, 5개 남음)
다시 실행하니 432→427개로 감소. 남은 5개(성수1가1동류 4건, 면목3.8동류 1건)를 진단해보니, 정규식이 두 가지 패턴을 못 잡고 있었음: ① 동 이름 중간에 이미 숫자가 낀 경우("성수1가1동"), ② 번호에 점이 낀 합성동("면목3.8동", 실제 공식명은 "면목제3.8동"). 정규식을 `(\D+)(\d+동)$`에서 `(.+?)([\d.]+동)$`로 완화해 해결.

### 10. 재검증 — 427개, 공식 목록 대조 unknown 0개
`resolve_precise_jibun.py`까지는 완전히 해결된 것으로 판단하고 다음 단계로 진행.

### 11. calc_surface_housing_cost.py, calc_consumption_burden_index.py 재실행 — 매칭 실패 재발 (42건)
표면 주거비는 577,761/577,761행 100% 계산 성공. 하지만 생활소비부담지수와의 결합에서 매칭 실패가 42건 재발. 원인이 세 갈래로 나뉨: ① 이미 아는 문제(신설동/용두동, 행정동 경계 재편), ② 새로운 패턴("가"가 낀 복합동: 금호2.3가동, 종로1.2.3.4가동 등), ③ "제" 문제 재발(가양제1동 등, 표면주거비 쪽엔 있는데 상권분석서비스 쪽엔 반영 안 됨).

### 12. 표본 규모 확인 — 파고들 가치 판단
42건이 실제로 몇 행에 영향 주는지 확인하니 약 13,000행(전체의 2.3%), 그중 표본 큰 동(성내2동 1,336건 등)도 섞여있어 무시하지 않고 계속 파기로 결정.

### 13. "가양1동" 정규화가 안 먹히는 이유 추적 — 근본 원인(전국 vs 서울) 발견
`normalize_dong_name("가양1동", official_set)`을 단독으로 테스트하면 정상적으로 "가양제1동"을 반환하는데, 실제 파이프라인에서는 반영이 안 됨. `debug_gayang.py`로 병합→중복제거→정규화→업종필터→건수필터→집계까지 단계별로 행 수를 추적해 **정규화 직후 단계에서 이미 실패**하는 것을 확인.

결정적 원인: `normalize_dong_name()`의 첫 번째 조건 `if candidate in official_set: return candidate`에서, "가양1동"(제 없는 형태)이 **서울이 아닌 다른 지역에 우연히 실제로 존재하는 행정동이라서** 이 조건에 걸려 즉시 반환되어 버렸고, 그 아래에 있는 "제 삽입해서 재확인" 로직이 아예 실행되지 않았던 것. 실제로 확인해보니 `"가양1동" in 전국_official_set == True`.

### 14. 2차 근본 수정 — 공식 목록을 서울로 한정
`dong_matcher.py`의 공식 목록 생성 로직에 `시도명 == "서울특별시"` 필터를 추가. 재실행 결과 공식 목록이 정확히 **426개**(서울 행정동 수와 일치)로 나왔고, `dong_matcher.py` 자체의 매칭 성공률도 100%로 확인.

### 15. calc_consumption_burden_index.py 재실행 — 29건으로 감소
서울 한정 공식 목록을 반영해 다시 돌리니 매칭 실패가 42→29건으로 감소. 남은 목록: 개포3동, 금호2.3가동, 면목제3.8동, 상계3.4동, 상계6.7동, 상일제1동, 상일제2동, 신설동, 용두동, 종로1.2.3.4가동, 종로5.6가동, 중계2.3동.

### 16. "가" 패턴 정규식 확장
"금호2.3가동", "종로1.2.3.4가동" 등 번호 뒤에 "가"가 붙는 복합동을 정규식이 못 잡는 문제를 확인하고, `[\d.]+동$`을 `[\d.]+가?동$`으로 확장 (`resolve_precise_jibun.py`, `calc_consumption_burden_index.py` 양쪽에 동일 반영).

### 17. 재실행했는데 결과가 그대로 — 코드 자체는 정상, 다른 원인 의심
수정 후 재실행했다고 보고받았는데 매칭 실패 건수가 29건으로 전과 동일하게 나옴. 정규식 로직을 독립적으로 재검증했더니 정상 작동을 확인. 눈으로 본 표기가 똑같아 보여도 실제로는 다른 문자가 섞여 있을 가능성을 의심하고, `repr()`로 문자 단위 비교를 제안.

### 18. 인코딩 손상 발견 — "." 이 "?" 로 깨짐
표면주거비 쪽("금호2.3가동")과 생활소비부담지수 쪽("금호2?3가동")을 `repr()`로 비교한 결과, 마침표(U+002E)가 물음표(U+003F)로 바뀌어 있는 것을 확인. 연도별 원본 파일을 하나씩 확인한 결과 **2026년 상권분석서비스 파일에서만** 이 손상이 발생한 것으로 확인 (2023~2025년 파일은 정상).

### 19. 3차 수정 — 물음표를 마침표로 되돌리기
숫자 사이의 "?"를 "."으로 되돌리는 전처리(`re.sub(r"(\d)\?(\d)", r"\1.\2", value)`)를 `calc_consumption_burden_index.py`에 추가. 재실행 결과 29→15건으로 감소.

### 20. "종로1.2.3.4가동"만 여전히 실패 — 연속 물음표 처리 한계
숫자가 4개나 이어지는 "종로1.2.3.4가동"은 여전히 실패. 원인: `(\d)\?(\d)` 패턴은 겹치지 않는 매칭만 처리해서, "1?2?3?4"처럼 물음표가 연달아 있으면 하나 걸러 하나씩만 고쳐지고 중간 하나가 남음. lookahead/lookbehind 방식(`(?<=\d)\?(?=\d)`)으로 교체해 숫자를 소모하지 않고 모든 물음표를 독립적으로 처리하도록 수정.

### 21. 최종 재실행 — 매칭 실패 12건 (5개 동)으로 수렴
재실행 결과 15→12건. 남은 5개 동을 각각 개별 조사:

- **신설동, 용두동** — 이미 알고 있던 행정동 경계 재편(2025.07 용신동 폐지) 이슈. 상권분석서비스의 과거 분기 데이터가 옛 이름("용신동")으로 남아있어 정규화로 해결 불가능한 시점 문제.
- **상일제1동, 상일제2동** — 원본 확인 결과 상권분석서비스는 2023~2026년 4개 파일 전체에서 "상일동" 하나로만 존재(분동 안 되어 있음). 반면 KIKmix 기준 우리 데이터는 "상일제1동"/"상일제2동"으로 분동. **정규화로 해결 불가능한, 두 데이터 소스 간 행정동 분류 기준 자체의 차이.**
- **개포3동** — 표면주거비 쪽엔 실제로 존재(단독다가구 15건, 오피스텔 233건). 그런데 상권분석서비스 2023~2026년 4개 파일 전체를 확인해도 "개포3동"이라는 명칭 자체가 없음(개포1·2·4동만 존재). 이상치 제거로 걸러진 게 아니라 **원본 데이터 자체에 이 명칭이 없는 구조적 결측** — 재건축 등으로 재편된 옛 행정동으로 추정.

### 매칭 실패 12건(1.1%)을 그대로 두는 이유 (제외의 논리적 근거)

이 12건을 억지로 임시방편(예: 상일동 값을 상일1·2동에 똑같이 복사, 개포3동을 인접 동 값으로 대체)으로 채우지 않고 결측인 채로 남겨둔 이유는 다음과 같다.

1. **원인이 전부 규명되었다.** 표기 문제(정규화 실패)가 아니라 ① 시점 문제(경계 재편), ② 소스 간 분류 기준 차이, ③ 원본 데이터의 구조적 결측이라는 것이 확인됐다. "우리가 못 찾은 것"이 아니라 "애초에 두 데이터가 서로 다른 방식으로 그 지역을 다루고 있다"는 사실이다.
2. **억지로 채우면 없는 정보를 지어내는 것과 같다.** 예를 들어 상일1동과 상일2동에 상권분석서비스의 "상일동" 통합 수치를 그대로 복사하면, 실제로는 존재하지 않는 "1동과 2동이 소비 수준까지 완전히 같다"는 가정을 데이터에 주입하는 셈이 된다. 이는 검증되지 않은 가정이라 위험하다.
3. **영향 규모가 방법론적으로 무시 가능한 수준이다.** 1,144개 조합 중 12건, 1.1%. 이미 3주차에 같은 원칙(행정동 경계 재편 영향 0.364%)으로 "영향이 미미하면 그대로 두고 문서화한다"는 기준을 세운 바 있다.

**방법론 문서에 넣을 문구**

> "표면 주거비와 생활소비부담지수를 행정동 기준으로 결합한 결과, 1,144개 조합 중 1,132개(98.9%)가 정상 매칭되었다. 매칭 실패한 12건은 5개 행정동에 국한되며 원인은 다음과 같다: ① 분석 기간 중 발생한 행정동 경계 재편(신설동·용두동), ② 데이터 소스 간 행정동 분류 기준 차이(상일1·2동 — 국토지리정보원 KIKmix는 분동 처리하나 서울시 상권분석서비스는 미분동), ③ 서울시 상권분석서비스 자체에 해당 명칭이 존재하지 않는 구조적 결측(개포3동 — 재건축 등으로 재편된 옛 행정동으로 추정). 세 원인 모두 매칭 로직의 결함이 아닌 원천 데이터 자체의 구조적 차이에서 기인하며, 전체 결과에 미치는 영향은 1.1% 수준으로 제한적이다."
