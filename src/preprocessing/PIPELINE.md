# 데이터 파이프라인 가이드 — 6개 파일이 하는 일

## 큰 그림 먼저

이 6개 파일은 하나의 목표를 위해 순서대로 실행되는 파이프라인이다.

> **국토부가 주는 원본 실거래가 데이터(법정동 단위, 9개 파일)를 → 우리 프로젝트가 필요로 하는 형태(행정동 단위, 청년 1인가구만 추출된 단일 파일)로 바꾸는 것**

국토부 데이터에는 딱 두 가지가 없다 — **(1) 행정동 정보**와 **(2) 임차인의 나이/가구원수 정보**. 이 두 개를 채워 넣는 과정이 파일 6개로 나뉘어 있는것. 하나의 거대한 스크립트로 안 만들고 쪼갠 이유는, 각 단계가 실패 지점이 다르고(인코딩 문제, API 실패, 네트워크 문제 등) 중간 결과를 저장해두면 문제가 생겨도 처음부터 다시 안 돌려도 되기 때문.

```
① housing_preprocessor.py   (9개 원본 → 1개 통합 파일)
        ↓
② dong_matcher.py           (법정동 텍스트 → 행정동 매칭 시도)
        ↓
③ resolve_ambiguous.py      (매칭 실패분 → 카카오 API로 1차 해결)
        ↓
④ juso_resolver.py          (카카오도 실패한 것 → juso.go.kr API로 2차 해결)
        ↓
⑤ resolve_precise_jibun.py  (연립다세대·오피스텔은 지번으로 더 정밀하게 재해결 + 최종 파일 생성)
        ↓
⑥ filter_youth_household.py (청년 1인가구로 추정되는 거래만 최종 추출)
```

---

## ① housing_preprocessor.py — 9개 파일을 1개로

**왜 필요했나**: 국토부 사이트에서 단독다가구·연립다세대·오피스텔 × 3개 연도 구간 = 9개의 csv를 따로따로 받아왔음. 분석하려면 이걸 하나로 합쳐야함.

**문제였던 점**: 파일 맨 위에 안내문구(공지사항 5~6줄)가 있어서 그냥 읽으면 깨졌고(`No columns to parse from file`), 인코딩도 파일마다 CP949/UTF-8이 섞여 있었다. 또 엑셀로 파일을 열어둔 채로 실행하면 `~$`로 시작하는 잠금파일까지 같이 읽혀서 에러가 났음.

**하는 일**: 각 파일에서 실제 데이터가 시작하는 행(`NO`가 첫 칸인 행)을 자동으로 찾아 그 위 안내문구는 건너뛰고, 인코딩도 자동으로 판별해서 읽음. `~$` 잠금파일과 이 파이프라인 자신이 만든 결과물(자기 자신을 다시 읽어버리는 "자기잠식" 버그도 있었음)은 제외하고, 진짜 원본 9개만 골라 하나로 합침.

**입력 → 출력**: 원본 csv 9개 → `전월세_실거래가_통합.csv`

---

## ② dong_matcher.py — 법정동 이름을 행정동으로 바꾸기 1차 시도

**왜 필요했나**: 우리 프로젝트는 서울시를 **행정동(주민센터 관할구역) 단위**로 나눠서 분석하기로 했는데, 국토부 데이터의 주소 필드(`시군구`)에는 **법정동(등기부에 쓰이는 옛 행정구역)** 이름만 있었다. 이 둘은 이름이 같을 때도 있지만 다를 때도 많아서, 국가가 공식 배포하는 법정동-행정동 코드표(KIKmix)와 매칭해야 한다.

**문제였던 점**: 법정동 하나가 인구 증가로 행정동 여러 개로 쪼개진 경우(예: "역삼동"이 "역삼1동"/"역삼2동"으로 분리)가 서울 전체 거래의 **73.2%**나 됐다. 코드표만 봐서는 이 중 어느 쪽인지 알 방법이 없었음.

**하는 일**: 법정동 이름으로 코드표와 매칭해서 행정동을 붙임. 분동된 경우엔 일단 후보 중 하나를 임시로 채택하고 `행정동_추정필요 = True`로 표시해서, 다음 단계에서 정밀하게 다시 풀도록 넘겼음.

**입력 → 출력**: `전월세_실거래가_통합.csv` + 법정동-행정동 코드표 → `전월세_실거래가_통합_행정동.csv`

---

## ③ resolve_ambiguous.py — 카카오 API로 도로명 기반 재확인

**왜 API가 필요했나**: 코드표는 "법정동 하나 → 행정동 몇 개"라는 매핑만 있지, "이 특정 거래가 그중 어디에 속하는지"는 알려주지 않음. 이건 실제 위치 정보(좌표 혹은 정밀 주소)가 있어야 풀 수 있는 문제라, 지도/주소 API가 필요.

**왜 카카오였나**: 카카오 로컬 API는 주소를 넣으면 법정동뿐 아니라 **행정동(`region_3depth_h_name`)까지 직접 반환**해주는 몇 안 되는 무료 API.

**문제였던 점 (그리고 왜 캐시가 필요했나)**: 732,419건을 전부 API로 조회하면 시간이 너무 오래 걸림. 하지만 자세히 보면, 같은 도로에 있는 거래는 수십~수백 건이 반복되고 있었음. 그래서 **거래 건이 아니라 "고유한 (시도, 구, 도로명) 조합"만 조회**하면 되고, 실제로 732,419건이 고유 조합으로는 6만 건 수준으로 줄었음. 처음엔 이마저도 요청 하나하나 순서대로 처리해서 시간당 1,800건 속도(전체 33시간 예상)가 나왔는데, 연결 재사용(Session)과 동시 요청(8개 병렬)으로 바꾸면서 훨씬 빨라졌음. 그리고 **100건마다 결과를 `kakao_cache.json`에 저장**해둬서, 중간에 노트북을 끄거나 에러가 나도 처음부터 다시 돌릴 필요 없이 이어서 진행할 수 있게 했음.

또 하나 — 처음엔 도로명에 법정동 이름까지 붙여서 검색했다가 실패율이 높았는데, 도로명주소 체계는 애초에 법정동과 무관하게 붙는 이름이라 오히려 법정동을 빼고 검색하니 성공률이 올라갔음. 그리고 API가 준 **첫 번째 결과를 검증 없이 그대로 믿는 것도 위험**하다는 지적이 있어서, 반환된 결과의 구(시군구)가 실제로 우리가 물어본 구와 일치하는지 확인하는 절차도 추가했음.

**입력 → 출력**: `전월세_실거래가_통합_행정동.csv` → 캐시 파일(`kakao_cache.json`) 갱신 + `전월세_실거래가_통합_행정동_보정.csv`

## ④ juso_resolver.py — 정부 공식 도로명주소 DB로 2차 보완

**왜 필요했나**: 카카오로 돌렸는데도 8,050개 조합이 실패했다. 실패한 도로명들을 확인해보니 "노량진로6가길"처럼 실제로 존재하는 정상적인 도로명이었다. 즉 데이터 문제가 아니라 **카카오맵(상업 지도)의 색인에 세부 골목길이 다 들어있지 않아서** 생긴 문제.

**왜 juso.go.kr이었나**: 이건 행정안전부가 관리하는 **대한민국 도로명주소 원본 DB**라서, 카카오맵에 없는 세부 이면도로까지 포함되어 있을 가능성이 훨씬 높다. 실제로 옵션 파라미터(`addInfoYn=Y`)를 켜면 행정동명(`hemdNm`)까지 바로 내려주는 걸 확인했음.

**하는 일**: 카카오에서 실패했던 8,050개(고유 조합)만 골라서 juso API로 재조회하고, 기존 `kakao_cache.json`을 직접 업데이트함. 여기서도 반환된 시군구(`sggNm`)가 우리가 물어본 구와 일치하는지 검증한 뒤에만 채택하도록 해둠.

**결과**: 8,050개 중 대부분이 추가로 해결되면서, 전체 매칭 성공률이 99.4%까지 올라갔음.

**입력 → 출력**: `kakao_cache.json`(카카오 실패분) → 같은 파일을 직접 갱신 (별도 출력 파일 없음, 다음 단계에서 다시 읽음)

---

## ⑤ resolve_precise_jibun.py — 연립다세대·오피스텔은 지번으로 한 번 더 정밀하게

**왜 필요했나**: 도로명 기반 매칭은 태생적 한계가 있음 — 하나의 도로가 실제로는 여러 행정동에 걸쳐 있는 경우, "그 도로 전체"에 대해 결과 하나를 그대로 복사해서 쓰기 때문에, 행정동 경계 근처의 거래는 잘못 분류될 위험이 남아있다.

**왜 이 파일이 단독다가구는 손 못 대고 연립다세대·오피스텔만 다루는가**: 단독다가구 데이터는 국토부 정책상 지번(번지)이 `1*`, `2*`처럼 마스킹되어 있어서 애초에 정밀한 위치를 알 방법이 없음. 반면 연립다세대·오피스텔은 다세대 건물이라 개인 식별 우려가 적어서 그런지, **본번·부번(지번)이 그대로 공개**되어 있었다. 지번은 한 필지를 정확히 가리키니까, "구+법정동+본번-부번"으로 조회하면 도로명 방식보다 훨씬 정밀한 매칭이 가능해졌다.

**하는 일**: 분동 애매 케이스 중 연립다세대·오피스텔만 골라 지번 기반으로 카카오 API를 재조회하고(435,574건 중 84.5% 성공), 최종적으로 **① 지번 정밀 결과 → ② 도로명 근사 결과(③에서 만든 것) → ③ 원래 텍스트 매칭 추정값** 순서로 우선순위를 매겨 최종 행정동을 확정한다. 결과적으로 분동 애매 케이스 732,419건 전부(100%)에 행정동이 채워졌다.

**입력 → 출력**: `전월세_실거래가_통합_행정동.csv` + `kakao_cache.json`(도로명) → `전월세_실거래가_통합_행정동_보정.csv` (최종 행정동 확정본)

## ⑥ filter_youth_household.py — 청년 1인가구로 추정되는 거래만 추출

**왜 필요했나**: 여기까지는 "어디에 있는 거래인지"를 확정하는 과정이었고, 이제 "이 거래가 청년 1인가구의 것으로 볼 만한지"를 걸러야 한다. 실거래가 데이터엔 나이 정보가 없어서, 면적·보증금·월세·주택유형 같은 간접 지표로 근사한다.

**적용 기준과 이유**:
- 면적 60㎡ 이하, 보증금 1억원 이하 → 청년 대상 정책(버팀목전세자금 대출 등)의 실제 상한값을 그대로 사용 (지역과 무관하게 고정된 값이라 절대 기준을 써도 무리 없음)
- 월세는 절대금액 대신 **상위 1% 분위수를 이상치 컷오프로 사용** → 절대금액(예: 60만원)을 쓰면 강남·마포처럼 시장가 자체가 높은 지역 표본이 통계적으로 소거되는 부작용이 있어서, 지역에 상관없이 "극단적으로 예외적인 값"만 걸러내는 방식으로 바꿈
- 전세 제외 → 청년층은 목돈 마련이 어려워 월세형이 다수

**하는 일**: 위 조건으로 걸러내며 단계마다 몇 건이 빠지는지 로그로 보여주고, 14㎡ 미만(고시원 등 비정형 주거 가능성)과 행정동별 표본 30건 미만(통계적 신뢰도 낮음)은 제외하지 않고 플래그만 남겨서 나중에 참고할 수 있게 했다.

**입력 → 출력**: `전월세_실거래가_통합_행정동_보정.csv` → `전월세_실거래가_청년1인가구.csv` (최종 분석 대상 파일)
